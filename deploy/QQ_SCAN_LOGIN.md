# QQ 扫码登录接入说明

这个项目本身是 **机器人业务框架**，不直接实现 QQ 官方登录流程。

要实现 **QQ 扫码登录**，需要配合一个支持 **OneBot v11 / HTTP API** 的 QQ 侧适配器运行。

## 推荐架构

```text
QQ 账号
  ↓ 扫码登录
QQ 侧适配器（如支持扫码登录的 OneBot 实现）
  ↓ OneBot HTTP API + 事件上报
qqbot-framework
```

## 你需要的能力

QQ 侧适配器需要至少满足：

1. 支持 **二维码扫码登录**
2. 支持 **HTTP API 发消息**
3. 支持 **反向 HTTP 上报消息事件**
4. 尽量兼容 **OneBot v11**

## 对接参数

机器人框架默认：

- 接收事件：`http://127.0.0.1:9000/onebot/event`
- 调用 QQ API：`http://127.0.0.1:5700`

因此你需要让 QQ 适配器满足：

### 1. 反向 HTTP 上报到

```text
http://127.0.0.1:9000/onebot/event
```

### 2. HTTP API 暴露在

```text
http://127.0.0.1:5700
```

如果端口不同，就改 `.env`：

```env
ONEBOT_API_BASE=http://127.0.0.1:5700
```

## 扫码登录的一般流程

不同 QQ 侧适配器名字、命令、UI 不同，但流程通常一致：

1. 启动 QQ 适配器
2. 终端日志 / Web 面板 / 本地页面里出现二维码
3. 用手机 QQ 扫码确认登录
4. 适配器保存登录状态
5. 机器人框架开始收到消息事件

## 登录状态持久化

无论你用哪种适配器，都建议把它的登录状态目录挂载到磁盘。

否则重启容器或重启服务后，很可能需要重新扫码。

## 安全建议

- 不要把 QQ 登录状态文件暴露到公网下载
- 不要把机器人的管理端口直接裸露到公网
- 给机器人配置 `QQBOT_OWNER_IDS`
- 生产环境建议只开放必要端口

## 重要说明

由于不同 OneBot/QQ 适配器项目维护状态、兼容性和风控情况经常变化，
**框架层面只负责兼容 OneBot 接口，不绑定单一登录实现。**

也就是说：

- 本框架：负责机器人业务逻辑
- QQ 适配器：负责扫码登录、收发消息、维持在线

## 最终效果

部署时你会得到：

- 一个负责业务逻辑的 `qqbot-framework`
- 一个负责 QQ 扫码登录和消息桥接的 OneBot 适配器

扫码登录成功后，这个框架就可以正常收发 QQ 消息。
